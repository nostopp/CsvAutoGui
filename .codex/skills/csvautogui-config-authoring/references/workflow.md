# 工作流

## 阶段 1：确认流程

目标：把自然语言需求整理成一份可审阅的流程方案。

要做的事：

- 检查 `config/` 下的目标项目目录
- 找出最接近的骨架
- 总结阶段和分支
- 尽量降低用户负担
- 只在真实阻塞点上追问

输出一份紧凑方案，至少包含：

- 建议的配置目录
- 建议的配置名
- 可能参考的骨架
- 阶段拆分
- 分支摘要
- 是否需要 `watchdog`
- 是否需要 `recovery.csv`
- 是否需要 stall 未解决通知或 `notify` 通知策略
- 预期输出文件

这一阶段不要写最终配置文件。

## 阶段 2：创建资源清单

流程方案确认后：

- 如有需要，创建配置目录
- 创建 `<config_name>_resource.csv`
- 预填用户必须采集的资源行

如果流程包含 config 级恢复：

- 把恢复流程需要的 pic/ocr 锚点一起纳入资源清单
- 不要把恢复资源拆到独立的编写清单

每一行都应该回答这几个问题：

- 要采集的是什么
- 它是 pic 还是 OCR 锚点
- 应该使用什么 alias
- 它服务于哪个阶段或分支

备注要写得足够清楚，让用户不必重新翻整段对话就能采集资源。

## 阶段 3：校验资源

当用户说资源已经采集完成时：

- 读取 `<config_name>_resource.csv`
- 校验每种资源要求的字段
- 校验 pic 图片文件真实存在
- 校验 alias 不重复
- 校验没有不支持的节点类型

如果资源不完整：

- 停止继续生成
- 只列出缺失或无效项
- 明确告诉用户需要补哪些采集或修哪些内容

## 阶段 4：生成配置

用已确认的流程和已验证的资源生成：

- 主流程
- 子流程
- 可选的恢复流程
- 可选的运行参数文件
- 仅在有充分理由时生成脚本文件

生成时保持和仓库现有模式一致：

- 能复用最近的骨架就优先复用
- 保持名称可读
- 避免不必要的 `script` 节点
- `recovery.csv` 的职责是把系统带回可从 `main.csv` 重新开始的状态
- `runtime.json` 只承载运行时策略，不承载流程本身
- recovery 阈值统一写到 `watchdog.recovery_watchdog`

## 阶段 5：校验最终输出

在宣布完成之前：

- 把生成的节点和流程方案交叉检查
- 把资源引用和资源清单交叉检查
- 确保文件名和跳转目标一致
- 如果生成了 `runtime.json`，检查字段名、层级和启用条件是否自洽
- 如果改了文件，提醒用户需要手动重载

## 知识更新

在产出可工作的配置后，检查这次是否形成了新的稳定项目规律。
如果形成了，就建议或直接更新 `references/projects-local/` 下对应文件中的一小段内容。

只记录：

- 可复用的骨架选择规则
- 稳定的 OCR 或 pic 锚点习惯
- 反复出现的恢复分支
- 命名或拆分约定
- 高价值坑点
