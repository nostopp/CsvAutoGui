# CsvAutoGui 配置编写

## 概述

把这个 skill 当作当前仓库内 CsvAutoGui 配置的约束式编写流程。
不要根据模糊的游戏或应用描述凭空设计自动化策略。这个 skill 的职责是把用户已经确认过的逻辑和用户已经采集到的资源，翻译成符合仓库约定的 `CSV / script / resource / runtime config` 输出。

先读 [references/core.md](references/core.md)。
再读 [references/workflow.md](references/workflow.md) 了解阶段式流程。
在最终落文件前读 [references/validation.md](references/validation.md)。

如果任务目标是已知项目族，例如 `game_a`，并且 `references/projects-local/` 下存在对应文件，也要一起读取。

## 工作模式

按四阶段工作流执行这个 skill。

1. 确认流程。
2. 提出配置路径并生成 `<config_name>_resource.csv`。
3. 校验已采集的 `pic/ocr` 资源。
4. 编写配置文件并执行配置级检查。

除非仓库里已经存在足够明确的结构和可复用资源，使得直接生成配置的风险很低，否则不要从模糊的自然语言需求直接跳到最终 CSV。

## 阶段 1：确认流程

从轻量的自然语言输入开始。
不要一上来就要求严格模板。

这一阶段要做的是：

- 识别目标项目，例如 `game_a` 
- 判断这是新流程、骨架变体，还是在现有流程上补分支
- 检查同项目目录下已有的配置
- 发现最接近的可复用骨架
- 把用户描述改写成一份紧凑的流程方案，供用户审阅

一份合格的流程方案通常应包含：

- 建议的配置名
- 建议的配置目录
- 可能参考的骨架
- 起始状态
- 主循环或结束条件
- 主要阶段，例如入口、战斗、结算、恢复
- 关键分支条件
- 纯 CSV 是否足够，还是应该上 `script`
- 是否需要 `watchdog`、`recovery.csv`、`runtime.json`
- 是否需要 stall 未解决通知或 `notify` 节点通知策略

除非用户在审阅流程方案后明确要求继续，否则这一阶段不要创建文件。

## 阶段 2：生成资源清单

在用户确认流程方案之后，提出配置目录和配置名建议。
如果用户想改，可以覆盖这两个建议。

确认后：

- 如有需要，创建配置目录
- 创建 `<config_name>_resource.csv`
- 只预填这次已确认流程真正需要的资源

这个清单文件服务于编写阶段，而不是运行时脚本调度。
其中不允许出现 `jmp` 资源。
只生成 `resource(pic;alias)` 和 `resource(ocr;alias)` 两类行。

使用能表达作用和阶段的语义化别名，例如 `entry_button`、`result_back_text`。

用 `备注` 列描述用户应该采集什么内容。
备注应写成采集说明，而不是实现说明。

如果 OCR 的目标文字已经明确，可以预填 `图片/ocr名称`。
如果 pic 目标图片还不存在，就把图片文件名留空，交给用户通过编辑器采集。

## 阶段 3：校验已采集资源

默认用户会通过现有 CSV 编辑器完成资源采集。

当用户返回并告诉你 `<config_name>_resource.csv` 已经采集完成时，检查：

- 每一行都是 `resource` 节点
- 这个清单文件里没有 `jmp` 资源
- alias 不重复
- `pic` 资源有文件名、有合法区域、且图片文件真实存在
- `ocr` 资源有目标文字、有合法区域
- 所有引用路径都没有越出当前配置目录

如果资源不完整，就不要进入最终配置生成。
只汇报缺失或无效的部分，并明确告诉用户还需要补采集什么。

如果后续生成配置时又发现缺少新的状态锚点，就回到这一阶段补充清单，而不是自行猜测。

## 阶段 4：生成配置文件

只有在流程已确认且资源充足的前提下，才生成真正的配置文件。

优先保持和仓库现有风格一致：

- 能复用现有骨架时优先复用
- 阶段天然分离时拆成子流程
- `script` 的使用要尽量少，并且理由充分
- 只有在流程确实需要统一恢复时才生成 `recovery.csv`
- 只有在需要配置 `watchdog`、stall 终态通知或 `notify` 节点通知通道时才生成 `runtime.json`
- 运行时 `*_resource.csv` 只在脚本真的需要时创建，不要和通用编写清单混用

常见输出包括：

- `main.csv`
- 子流程 CSV，例如 `check.csv`、`pending.csv`
- `recovery.csv`
- `runtime.json`
- `script.py`
- 仅在生成的脚本确实需要时才创建运行时 `*_resource.csv`

结束前应用 [references/validation.md](references/validation.md) 中的检查项。

## 项目知识

项目特定知识放在 `references/projects-local/` 下。
这些文件本来就是本地化沉淀，允许被 git 忽略。

如果存在对应项目文件：

- 在共享 reference 之后读取它
- 用它来细化骨架选择、资源命名和分支结构
- 把它当作实现知识，而不是业务说明文档

如果这次任务产出了新的稳定规律，在确认配置可用后，建议更新对应的本地项目知识文件。

## 边界

这个 skill 不是全自动的策略设计器。

适合用它做：

- 把已确认的逻辑翻译成 CsvAutoGui 结构
- 发现可复用骨架
- 脚手架化资源采集
- 校验采集好的资源
- 生成并校验配置文件

不要用它做：

- 根据一句业务描述凭空发明未知 UI 状态
- 捏造截图或 OCR 锚点
- 在依赖未验证 UI 标记的情况下跳过资源采集
